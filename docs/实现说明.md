3. 记录按时间倒序维护，最新记录放最上面。
4. 若本次未运行测试，必须明确写出原因。

## 2026-03-16 | V3-011 新增基金后一键更新“假失败”修复（前端超时）

### 背景
- 用户在新增基金后点击“一键更新”，页面提示“更新失败”。
- 复现发现后端 `POST /pipeline/run` 实际成功，但耗时约 60~70 秒，超过前端固定 30 秒超时，导致误报失败。

### 改动文件
- `apps/web/dashboard.py`

### 操作步骤
1. 给请求工具函数增加可配置超时参数：
   - `safe_get(..., timeout=...)`
   - `safe_post(..., timeout=...)`
2. 对长耗时操作单独放宽超时：
   - 一键更新 `/pipeline/run`：`timeout=180`
   - 仅重算建议 `/decision/run`：`timeout=120`
3. 在首页提示中补充时长预期：
   - 新增基金后可能需要 `30-90 秒`（含 DeepSeek 复核阶段）。
4. 清理重复 API 进程并统一重启 API + 看板，避免多实例导致行为不一致。

### 功能说明
- 一键更新不再因为“前端先超时”而误提示失败。
- 用户能在 UI 里提前知道长耗时属于正常现象，而非系统异常。

### 验证结果
- 语法检查通过：`python3 -m py_compile apps/web/dashboard.py`
- 复现调用返回成功：`POST /pipeline/run` 返回 `200`，包含 `ingest_run_id/decision_run_id/recommendations`
- 真实耗时验证：约 `69.8s`（此前 30s 前端超时会误判失败）
- 服务可访问：
  - `http://127.0.0.1:8000/health` -> `ok`
  - `http://127.0.0.1:8501/` -> `200 OK`
  - `http://localhost:8501/` -> `200 OK`

### 后续事项
- 可进一步把一键更新改为异步任务 + 轮询进度条，避免长请求阻塞。
- 可在“高级信息”里展示本轮耗时分解（采集/决策/日报各阶段）。

## 2026-03-12 | V3-009 修复 005052 当日净值延迟显示

### 背景
- 用户反馈 `摩根标普港股通低波红利指数C（005052）` 看板中的“最近表现”仍显示昨天的涨跌。
- 排查发现并非基金官方未更新，而是本地行情抓取逻辑优先命中了较慢更新的来源，且备用来源参数不稳定。

### 改动文件
- `packages/common/market_data.py`
- `tests/test_market_data.py`

### 操作步骤
1. 排查 `005052` 的两个东方财富来源：
   - `pingzhongdata`：最新停在 `2026-03-11`
   - `lsjz`：已更新到 `2026-03-12`
2. 修复短窗口请求误判：
   - 去掉“少于 20 条就视为失败”的限制，避免 `days=10` 这类请求错误回退 `proxy`。
3. 修复来源选择策略：
   - 同时请求 `pingzhongdata` 与 `lsjz`
   - 优先返回“最后交易日更晚”的那一个来源
4. 修复 `lsjz` 参数稳定性：
   - 去掉 `startDate/endDate`
   - 将 `pageSize` 限制在稳定可用区间（最大 `200`）
5. 新增测试，覆盖“优先选择更新日期更晚的来源”和“短窗口不回退 proxy”。

### 功能说明
- `005052` 这类基金现在会优先使用当日已更新的净值来源，不再被 `pingzhongdata` 的慢更新拖住。
- 短周期图表和涨跌计算不再因为窗口较小而错误回退到代理数据。

### 验证结果
- 本地函数验证：
  - `pingzhongdata` -> `2026-03-11`
  - `lsjz` -> `2026-03-12`
  - 最终选择 -> `eastmoney_lsjz / 2026-03-12 / 1.2811`
- 自动化测试通过：
  - `pytest -q` -> `15 passed`

### 后续事项
- 可继续为不同基金类型建立来源优先级画像（QDII、港股、场外指数基金等）。
- 可在页面中直接展示“净值日期”，避免用户误把 T+1 与数据故障混淆。
